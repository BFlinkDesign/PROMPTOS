# Task Report

## Outcome

State the verified result first. If the result is partial, say so before listing
implementation details.

## Files changed

List each changed path and its purpose. Separate required task changes from
optional or deferred work. Note any dirty worktree or unrelated changes that were
preserved.

## User-visible behavior

State what changed, what intentionally did not change, and any platform behavior
that remains unverified.

## Verification

| Command / interaction | Environment | Result |
| --- | --- | --- |

Include exact commands, target-host interactions, screenshots, logs, hashes, or
other evidence when applicable. Do not claim tests passed unless they were run
and observed.

## Acceptance criteria

- [x] Passed criterion with evidence.
- [ ] Failed or unverified criterion with reason.

## Authority

Name the source-of-truth files, issue, plan, or acceptance criteria used. Note
any conflicts or instructions treated as evidence rather than authority.

## Blockers

List missing-source-of-truth, inaccessible-target-host, stale-evidence, failed
gates, approval needs, or false-green-test gaps.

## Risks and limitations

Be explicit about target-platform behavior not exercised, data/source
assumptions, migration risk, rollback limits, or known debt.

## Follow-up

Name one focused next checkpoint, if needed, and the fallback if that checkpoint
cannot be run.
